### 1.x Implementation note: also converted `${{ }}` shell interpolations to env vars in the two edited steps

The `Set release version` and `Update develop branch` steps previously interpolated `${{ inputs.release-version }}`, `${{ steps.validate.outputs.release_branch }}`, and `${{ steps.next-version.outputs.next_snapshot_version }}` directly inside `run:` shell commands. Per [GitHub's workflow injection guidance](https://github.blog/security/vulnerability-research/how-to-catch-github-actions-workflow-injections-before-attackers-do/), these expressions are substituted at YAML-render time and can introduce shell injection if any source is attacker-controlled. The values here are validated upstream (regex `^[0-9]+\.[0-9]+\.[0-9]+$` etc.), so the risk is theoretical, but the local pre-tool security hook flags the pattern. Lifting the values into a step `env:` block and referencing them as `$RELEASE_VERSION` / `$RELEASE_BRANCH` / `$NEXT_SNAPSHOT_VERSION` resolves both the hook and the underlying class of risk, with no behavior change. Other steps in `_release_common.yml` retain their existing `${{ }}` style and were not touched.
